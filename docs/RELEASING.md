# Releasing

All eight packages version in lockstep. A release is: bump → tag → push;
CI publishes to PyPI via OIDC Trusted Publishing (no stored tokens).

## One-time PyPI setup (Trusted Publishing)

Each PyPI project must trust this repo's `release.yml`. For **every**
package (`semql`, `semql-mcp`, `semql-erd`, `semql-validate-db`,
`semql-engine`, `semql-introspect`, `semql-auth`, `semql-prompt`):

1. PyPI → the project → *Manage* → *Publishing* → *Add a pending publisher*
   (or add to an existing project).
2. Fill in:
   - **Owner**: `semql-io`
   - **Repository**: `semql`
   - **Workflow name**: `release.yml`
   - **Environment**: `pypi`
3. In this repo's GitHub settings, create an **Environment** named `pypi`
   (Settings → Environments). Optionally add required reviewers so a human
   approves each publish.
4. Enable the publish gate: Settings → Secrets and variables → Actions →
   **Variables** → add `ENABLE_PYPI_PUBLISH` = `true`. Until this variable
   is `true`, the `publish` job is skipped and nothing reaches PyPI (builds
   and GitHub Releases still happen). Delete/flip it to pause publishing.

No API tokens are stored anywhere once this is set up. The legacy
token-based `just release` path (sops/age) still works as a fallback.

## Cutting a release

```sh
just bump minor          # or patch / major — rewrites all versions + sibling ranges
# review the diff, update CHANGELOG.md
git commit -am "Release vX.Y.Z"
git tag vX.Y.Z
git push && git push --tags
```

Pushing the tag triggers `.github/workflows/release.yml`:

1. **build** — `uv build` all eight packages, `twine check` the metadata.
2. **publish** — upload to PyPI via `pypa/gh-action-pypi-publish` (OIDC).
   Runs only on a tag push and only when `ENABLE_PYPI_PUBLISH` is `true`.
3. **github-release** — create the GitHub Release with autogenerated notes
   and the built wheels/sdists attached.

## GitHub-artifacts-only release (no PyPI)

Trigger the workflow manually (Actions → Release → *Run workflow*, or
`gh workflow run Release -f tag=vX.Y.Z`) to build and publish a GitHub
Release with the artifacts, without touching PyPI. The tag is created at
the chosen commit if it doesn't already exist.

## Manual fallback

The `Justfile` still carries the token-based recipes (`just release`,
`just release-staged`, `just release-github`) for publishing from a
workstation if CI is unavailable. They decrypt the PyPI token from
`secrets.enc.yaml` via sops/age.
